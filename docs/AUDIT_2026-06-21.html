#1

4 sequential ML HTTP calls replace 1 — live signal endpoint can block for 120s confirmed

signal-engine/src/generators/signals.py line 1763
What changed: generate_all_signals() now calls _fetch_ml_data(symbol, sk) 4× sequentially (SHORT → SWING → LONG → GROWTH). Each call tries 3 endpoints with a 10s timeout each.

Failure path: GET /signals/{symbol}?live=true (routes.py:3090) calls this synchronously. When ML service is unreachable, each style exhausts 3×10s = 30s before giving up. Total: 4×30 = 120s. The api-gateway proxy timeout is 120s — live signal requests time out instead of gracefully falling back to TA-only.

Background bulk persist is unaffected (runs as background task). Only live signal requests (?live=true) are impacted.

Fix: Use ThreadPoolExecutor to fetch all 4 styles in parallel. Worst-case drops from 120s to 30s.

#3

Falsy check `if _atr_14` stores None for exactly-zero ATR plausible

signal-engine/src/generators/signals.py line 1824
Bug: reasons["atr_14"] = round(_atr_14, 4) if _atr_14 else None is falsy for both None AND 0.0. A stock with zero intraday range on all 14 recent bars (e.g., HK micro-cap or halted stock) would have _atr_14 == 0.0, but it's stored as None.

#4

ATR-based game plan silently breaks 2:1 R:R when stop is wide and target is capped plausible

decision-engine/src/api/core/aggregator.py lines 53-54
Logic: rr_target = live_price + 2.0 × (live_price − stop) then take_profit = min(rr_target, live_price × target_pct).

Failure: SWING has target_pct = 1.12. If ATR stop is 7% below entry, rr_target = +14% → capped at +12%. Effective R:R = 12/7 = 1.71:1, not 2:1. The gap worsens as stop distance grows beyond 6%. The game plan dict returned has no R:R field — callers see stop+target that appear internally consistent but the 2:1 guarantee is silently broken.

Fix: Add rr_ratio = round((take_profit - live_price) / max(live_price - stop, 0.001), 2) to the returned dict so the UI can surface when the actual ratio deviates from the target.
#5

asyncio.get_event_loop() inside running async context — DeprecationWarning now, crash on Python 3.12 plausible

research-engine/src/api/routes.py lines 1477, 1488
Code: Two occurrences of loop = asyncio.get_event_loop() inside async def generate_research().

Python 3.11 (current): Emits DeprecationWarning but works — returns the running loop.

#6

In-memory cache miss race — concurrent requests for same symbol both trigger full AI generation plausible

research-engine/src/api/routes.py line 1449
Trigger: A BUY signal fires for a popular symbol. Multiple users click the Research button within milliseconds (email alert link, opportunities page refresh). Both requests call _cache.get(sym), both miss, both proceed through asyncio.gather data fetching, both call Claude/DeepSeek. Second write overwrites first. Each generation costs ~$0.05–$0.15 in API tokens.

Frequency: Happens reliably on every BUY email if multiple subscribers open the link simultaneously — exactly the highest-traffic moment.

Fix: Per-symbol asyncio.Lock — first waiter computes and caches; second waiter acquires lock, finds warm cache, returns immediately.
