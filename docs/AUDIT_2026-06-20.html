AI Signal Engine Audit

Path: services/signal-engine/src/generators/signals.py

Architecture

Signal generation uses a 4-pillar TA score (Trend 35% · Momentum 25% · Volume 20% · Structure 20%), fused with ML probability via a dynamic weight (0–75% based on AUC). Multi-timeframe confirmation (weekly RSI/MACD/trend) boosts or compresses the result. 12+ sequential suppression filters apply per-style multipliers. Four style profiles (SHORT/SWING/LONG/GROWTH) share one data-fetch pass with different thresholds.

Critical Bugs

RSI Divergence Hard-Disabled — Dead Weight in TA Calibration
signals.py:852
CRITICAL
rsi_divergence = "none" is set unconditionally — the detection logic was disabled when a bug was found (argmax comparing timing indices instead of RSI levels). But "rsi_divergence_bullish" remains in REASONS_MAP and TA_WEIGHTS. Its weight is drawn from the calibration but its contribution is always zero. This silently corrupts the TA score calibration by allocating probability mass to a feature that never fires.
Remove rsi_divergence_bullish from REASONS_MAP and TA_WEIGHTS; redistribute its weight to MACD momentum. Then rewrite divergence detection to compare RSI values at confirmed swing highs/lows (not argmax index positions).
SWING Pre-Earnings Compression Makes BUY Mathematically Impossible
signals.py:1093
CRITICAL
SWING earnings compression thresholds: {2: 0.50, 5: 0.75, 10: 0.90}. SWING buy_threshold: 0.67. A borderline signal at fused = 0.70 with earnings in 2 days: 0.70 × 0.50 = 0.35 — fails threshold. The beat-rate scaling (up to ×2.0) restores to 0.70 at best, never generating new BUY. Reliable earnings beaters are silently blocked from SWING signals in 2-day windows.
Raise SWING earnings thresholds to {2: 0.65, 5: 0.85, 10: 0.95}. Validate against 90 days of SWING outcomes in pre-earnings windows.
Stale Price Tolerance: Code 5 Days vs Comment 3 Days
signals.py:1707
HIGH
if days_old > 5 marks data stale. Comment says "3 calendar days." Signals can be computed on 4–5 day old bars and published without the 0.60× suppression being applied.
Change to if days_old > 3. Add a log warning at days_old > 1 so stale data is visible before hitting the threshold.
ML Model Age Not Tracked — Stale Models at Full Weight
signals.py:242
HIGH
Signal-engine uses ml_test_auc to determine ML weight (up to 0.65–0.75× of fusion). No check on when the model was retrained. A 45-day-old model with AUC 0.72 gets full weight as one retrained yesterday. Market regimes shift; stale models may confidently predict the wrong direction.
Add model_retrain_date to ML prediction response. Compute model_age_days in signal-engine; apply exponential decay: full weight at 0–7d, 80% at 14d, 50% at 30d, 20% at 60d. Log when age > 21 days.
Weekly Gate Post-Compression Cap Cannot Be Rescued
signals.py:1654
HIGH
The 0.40× weekly gate (weekly_rsi ≤ 38 AND trend down) fires after the compression cap. All other filters respect the cap (min 55% floor for SWING), but the weekly gate bypasses it — a signal at the cap floor is then crushed to ~22% of original. Undocumented; may over-suppress during temporary RSI dips in otherwise healthy stocks.
If RSI dip is recent (<5 bars), apply 0.65× instead of 0.40×. If confirmed weekly downtrend >20 bars, keep full 0.40×. Store weekly_gate_reason (brief_dip vs extended_downtrend) in reasons.
No Conviction Gate — Sequential Filters Replace Coherent Scoring
signals.py:1244
MEDIUM
The original 8-layer Conviction Gate has been replaced by sequential probability multipliers. No unified gate boolean exists. The frontend Gate UI (FULL/NEAR/FAILED) reconstructs state from the reasons dict on the fly — it may not match backend suppression state if reasons format changes.
Add conviction_layers dict to signal reasons (PASS/SOFT/FAIL per check + actual value) and gate_score (e.g. 6/8 PASS). Frontend reads directly rather than reconstructing.
S/R Levels Not Weighted by Touch Count
signals.py — _sr_context()
MEDIUM
S/R context applies ±0.05 adjustment equally to all levels. A level touched once gets the same weight as one touched 5 times. Institutional levels (tested repeatedly) carry far more significance.
Cluster nearby levels within 0.5%. Weight by touch count: 1 touch=±0.03, 2=±0.05, 3+=±0.08.

ML Prediction Pipeline Audit

Path: services/ml-prediction/src/

Architecture score: 9/10 — Production-grade with excellent anti-leakage and overfitting controls. Zero look-ahead bias confirmed. TimeSeriesSplit(5) + four-way temporal split (70/80/90/100%) prevents all data contamination. Probability calibration on isolated holdout set. Blended sample weights (recency × balanced class). Overfitting detection (CV-test AUC gap > 0.10 triggers suppression). Multi-model ensemble XGBoost (40%) + LightGBM (35%) + RandomForest (25%) with dynamic weighting.
Documentation discrepancy: Code lists 22 features but actual FEATURE_COLUMNS contains 46 features: 28 technical (momentum, volatility, trend, oscillators, volume), 8 macro (SPY returns, VIX, regime flags), 10 fundamental (revenue growth, ROE, FCF yield, P/B, D/E).

Issues Found

ML Output Never Flows into Signal Generation
market-data/scheduler.py · signal-engine/signals.py:242
CRITICAL
Signal-engine reads ml_probability from a Redis cache key it expects market-data to populate. But market-data's scheduler has no task calling POST /ml/predict_ensemble_three. The cache is never populated. Signal-engine's ML fusion may be using empty or stale data for most symbols — effectively running as TA-only for the majority of the stock universe. The ML service is trained, tuned, and running but disconnected from the main signal pipeline.
Add daily batch task in market-data scheduler: for each tracked symbol × 4 styles, call POST /ml/predict_ensemble_three and write result to ml:pred:{symbol}:{style} in Redis (TTL 28h). Signal-engine reads from this key. Or: have signal-engine call ml-prediction directly (async, 5s timeout) with Redis fallback.
LSTM Model Incomplete — Returns 0.5 for Short Histories
ml-prediction/src/models/lstm.py
MEDIUM
The LSTM model stub returns 0.5 (coin flip) when sequence length < minimum. It is registered in the model registry but contributes noise to any ensemble that includes it. No LSTM training pipeline exists; it's a placeholder.
Remove LSTM from the model registry entirely until fully implemented. Its 0.5 return dilutes ensemble signal on short histories.
Missing Market Regime Features in ML Training
ml-prediction/src/features/builder.py
MEDIUM
Features include SPY returns and VIX level but not: yield curve spread (10Y-2Y), credit spreads, sector momentum relative to market, or trend regime classification (up/down/range-bound). These macro context features are strong predictors of which signals are reliable vs unreliable.
Add: (1) yield_curve_spread = 10Y - 2Y treasury spread, (2) sector_rs_rank = sector ETF RS percentile, (3) regime_class from the decision-engine regime endpoint (once exposed), (4) vol_ratio_5d20d (already computed in ranking-engine, add to ML features), (5) earnings_days_away and beat_rate.
No Auto-Retraining When Model Goes Stale
ml-prediction/src/api/routes.py · market-data scheduler
MEDIUM
Models are retrained on-demand via POST /ml/tune_all. No automatic trigger exists when model_age exceeds a threshold. In production, models can go 30–60+ days without retraining if the operator forgets the manual tune.
Add weekly check in market-data scheduler: if model_retrain_date for any style > 21 days, trigger POST /ml/tune_all?styles={stale_styles} automatically. Notify via log + admin alert.
Hardcoded Train/Test Split Ratios
ml-prediction/src/training/trainer.py
LOW
Four-way temporal split is hardcoded at 70/80/90/100%. These should be configurable, especially as more data accumulates. The calibration set (90–100%) may shrink if the total dataset grows and 10% becomes too small for isotonic regression.
Move split ratios to ml_config.json alongside other tuning parameters. Add minimum calibration set size check (at least 200 samples).
Survivorship Bias Adjustment Is Intuition-Based, Not Validated
ml-prediction/src/training/trainer.py
LOW
A +3pp precision adjustment for survivorship bias is applied to performance metrics. This is noted in comments as intuition-based. The actual survivorship bias impact depends on the proportion of delisted stocks in the training universe, which is not measured.
Track SignalOutcome (actual forward returns) per symbol in production. Use real outcomes to measure and calibrate the adjustment empirically rather than applying a fixed +3pp.
What's working well: Zero look-ahead bias across all 46 features confirmed. TimeSeriesSplit prevents temporal leakage. Probability calibration (IsotonicRegression ≥300 samples, LogisticRegression <300, graceful fallback) is industry-standard. Optuna tuning with MedianPruner and 9-parameter search space is thorough. Atomic persistence via tempfile + os.replace() prevents model corruption on restart. Overfitting detection via CV-test AUC gap is a strong guard. Multi-model ensemble with dynamic weighting and agreement detection is sophisticated.

Regime Engine Audit

Path: services/decision-engine/src/api/core/regime.py (embedded — no standalone service)

There is no regime-engine service. Regime detection is embedded exclusively inside the decision-engine with a local 4-hour cache. No HTTP endpoint exposes it. No other service can access it. Signal-engine, research-engine, and portfolio-optimizer each implement their own simplified regime proxies — creating inconsistent regime state across services.
Regime Logic Locked in Decision-Engine — Not Accessible by Any Other Service
decision-engine/src/api/core/regime.py:24–198
CRITICAL
The regime computation (lines 61–198) classifies markets as bull/bear/choppy/risk_off/neutral using SPY EMAs, VIX level, IWM/MDY breadth — comprehensive and correct. But it lives inside decision-engine with no HTTP endpoint. Signal-engine checks Fear & Greed separately. Research-engine has no regime context. Portfolio-optimizer has no regime-aware position limits. The system can simultaneously show "Bull" in research and "choppy" in signal suppression filters.
Add GET /decide/regime?market=US|HK to decision-engine. Cache result in Redis under regime:{market} with 30-min TTL when VIX > 20, 4h otherwise. Signal-engine replaces Fear & Greed check with this endpoint. Research-engine quotes regime in reports. Portfolio-optimizer adjusts position limits by regime state.
Signal Engine and Decision Engine Have Inconsistent Regime State
signals.py — market_regime fetch vs regime.py
HIGH
Signal-engine applies high_vol_regime (Fear & Greed < 30) and market_breadth (<40%) independently. Decision-engine uses a 5-state regime based on SPY EMAs, VIX, and small-cap breadth. A signal can be generated under "normal" conditions and then blocked by decision-engine operating in "risk_off" — confusing to diagnose.
Once regime endpoint is exposed: signal-engine calls it and stores market_regime in reasons dict. All downstream decisions (alerts, paper trading, game plan) reference the same regime.
Portfolio-Optimizer Has No Regime-Aware Position Limits
portfolio-optimizer/src/api/routes.py
MEDIUM
All 4 optimization methods (Mean-Variance, Risk Parity, HRP, AI Allocation) have no regime check. In bear or risk_off regimes, the optimizer may still recommend full allocation to volatile stocks.
After exposing regime endpoint: apply regime multiplier to max position limits: bull=1.0, choppy=0.75, bear=0.60, risk_off=0.50. Return applied multiplier in optimization response.
4-Hour Regime Cache Misses Intraday VIX Spikes
regime.py:14
MEDIUM
VIX can spike 30%+ intraday. A spike from 15 to 35 at market open may not change the regime classification for up to 4 hours. Signals generated during this window use pre-spike (bull) thresholds.
Conditional cache TTL: ttl = 30*60 if vix > 20 else 4*60*60. Implement as part of the shared Redis cache once regime endpoint is exposed.

Decision-Making Pipeline Audit

Conviction gate · Game plan · Paper trading · Kelly sizing · Opportunities · Alerts

Critical Bugs

Paper Trading Bypasses Conviction Gate — Uses Different _should_enter() Logic
market-data/src/services/scheduler.py:516–673
CRITICAL
The conviction gate (_is_conviction_buy()) in the scheduler has 5 layers and produces verdicts for decisions. But paper trading uses a separate _should_enter() function that reimplements gate logic inline with different thresholds. Paper trades can enter positions that the conviction gate explicitly rejected. Decision-engine is run in "shadow mode" and its verdicts are logged but not used by paper trading.
Replace _should_enter() in paper trading with a call to the decision-engine verdict. If decision-engine verdict is BUY (not SKIP/BLOCKED/HOLD), enter. Shadow mode should not be shadow — it should be the authority. Remove the inline reimplementation.
STYLE_PARAMS Mismatch — Decision Engine vs Paper Trading Have Different Stop %
decision-engine/src/ vs market-data/src/services/scheduler.py
CRITICAL
Decision engine SWING stop: 8.8%. Paper trading SWING stop: 5.5%. Same style, different risk parameters. Game plans generated by decision-engine are inconsistent with what paper trading actually executes. Win rate statistics from paper trading cannot be applied to decision-engine recommendations.
Create a single shared STYLE_PARAMS config (JSON or common/ module). Both services import from the same source. Any parameter change updates both simultaneously.
Game Plan Uses Fixed % Stops — No ATR or S/R Levels
decision-engine/src/api/routes.py
HIGH
Stop loss in the decision-engine game plan is a fixed percentage from entry price (SWING: 8.8%). No ATR-based stops (which adapt to each stock's volatility) and no support/resistance level alignment (which is where institutional players actually place stops). Paper trading uses ATR-based stops — creating another STYLE_PARAMS mismatch and poor stop placement in practice.
Use ATR-based stops: stop = entry - (2.0 × ATR_14). Also check nearest support level — use the higher of ATR-stop or nearest support (within 5%) as the stop price. This ensures stops are placed at meaningful technical levels.
Kelly Criterion Not Used — Fixed 1% Risk Per Trade
decision-engine/src/api/routes.py · scheduler.py
HIGH
Position sizing uses fixed 1% risk per trade throughout. Kelly Criterion is referenced in the frontend (ML panel) but not computed by any backend service. For a strategy with 55% win rate and 2:1 R:R, optimal Kelly is ~10%, half-Kelly is 5% — systematically undersizing winners. Historical closed trades are available in paper trading but not used for Kelly calibration.
Implement Kelly per-style from closed paper trades: kelly_f = win_rate - (1 - win_rate) / rr_ratio. Use quarter-Kelly for live sizing. The signal-engine already computes confidence — use it to scale position: high-confidence BUY at 3% risk, medium at 2%, low at 1%.
Opportunities Page Does Not Filter by Conviction Gate
frontend/src/pages/opportunities.tsx
MEDIUM
Opportunities tabs (Swing, Short-Term, Long-Term, Growth, Confluence) rank stocks by signal and K-Score but do not filter by conviction gate status (FULL/NEAR/FAILED). A stock with Gate FAILED appears identically to one with Gate FULL — users don't know which are actually actionable.
Add gate status to the opportunities API response. In the Confluence tab especially, filter to Gate FULL only. In other tabs, show gate status badge (FULL / NEAR / —) on each card and allow filtering.
Conviction Gate Not Exposed to Frontend — Users Can't See Which Layers Failed
scheduler.py:516–673 · stock/[symbol].tsx
MEDIUM
The conviction gate runs 5 checks (K-Score, uptrend, RSI bounds, MACD momentum, OBV trend) in the scheduler but does not return which layers passed or failed. The stock detail page shows FULL/NEAR/FAILED but cannot show WHY. Users have no way to understand what needs to change for a NEAR to become FULL.
Return conviction_layers in signal response: each layer stores PASS/FAIL + actual value (e.g., {"rsi": {"status": "PASS", "value": 58.3, "threshold": [35, 75]}}). Frontend renders a mini gate breakdown showing each layer's status — turns the Gate from a black box into a diagnostic tool.
Alert Emails Don't Show Gate Status or Suppression Reason
market-data/src/services/scheduler.py — check_signal_alerts()
LOW
Signal change emails (▲ Upgrade / ▼ Downgrade) don't include the gate status (FULL/NEAR/FAILED) or key suppression conditions. Users can't act on an alert without opening the app to check if the trade is actually clean.
Include gate status and top 2 suppression conditions (if any) in the email body: "SWING BUY — Gate: NEAR (Weekly misalignment, RSI 71 — near overbought). Confidence: 64%." This alone saves a round-trip to the app for each alert.
What's working well: Decision-engine async fan-out (signal + research in parallel) prevents timeouts. Hard reject logic (regime, research gating, portfolio limits) is comprehensive. Position scoring (4-point scale) is clear and auditable. Service JWT auth between decision-engine and its upstreams is correct. Paper trading tracks trade history and generates closed trade statistics.
